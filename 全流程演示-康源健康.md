## Step 0：架构总览

### v1.8.0 三层API架构

```
┌──────────────────────────────────────────────────────────┐
│  顶层 API（4个入口）                                       │
│  run_full_diagnosis / run_roi_only                       │
│  run_store_diagnose / run_pitfall_check                  │
├──────────────────────────────────────────────────────────┤
│  中层 Pipeline + Task（流程编排）                          │
│  DataValidation → Score → Scenario → ROI → Roadmap       │
│  + 状态持久化 + 数据一致性校验 + Session管理               │
├──────────────────────────────────────────────────────────┤
│  底层 引擎脚本 + 本体知识库                                │
│  calculate_*.py  +  ontology.py（8大知识模块）             │
└──────────────────────────────────────────────────────────┘
```

### 诊断6步流程

```
Step 1 信息收集 → Step 2 动态评分 → Step 3 ROI分析
     ↓              ↓              ↓
Step 6 门店诊断 ← Step 5 路线图 ← Step 4 场景推荐
```

> **【改进点】** v1.8.0 借鉴 cognee 知识图谱框架的5大设计思想：
> 1. **三层API架构**：顶层4函数 → 中层Pipeline → 底层引擎+本体，职责清晰
> 2. **Pipeline + Task 编排**：6个Task按依赖关系自动执行，支持跳过/回溯
> 3. **状态持久化**：SessionState 可保存为JSON，支持中断续答
> 4. **真相子空间**：数据一致性校验，4条规则自动检查异常
> 5. **本体知识库**：ontology.py 统一管理8大知识模块，所有脚本口径一致

## Step 1：信息收集 - 用户发起咨询

### 1.1 执行过程

#### 用户侧（模拟）

> **用户**：我们是一家做保健品+健康服务的公司，有28家门店，300多员工，年营收大概1.2亿。现在感觉人力成本越来越高，老客户复购率在下降，门店服务质量也参差不齐，想看看AI能不能帮我们解决这些问题？大概需要投多少钱？多久能回本？

#### Skill 响应（v1.8.0 方式）

**第一步：识别触发 + 确认行业类型**

> **AI诊断助手**：您好！感谢您的咨询。我是健康行业AI落地诊断助手，将帮您系统评估企业的AI落地可行性、推荐场景和ROI测算。

> 1. 您的公司全称是？（用于报告抬头）

**第二步：26题问卷分4批收集 + 每批确认**

## Step 2：五维度动态评分

### 2.1 执行过程

信息收集完成后，调用 `scripts/calculate_score.py` 脚本进行五维度评分。

**输入数据**（从问卷答案映射到5维度分值）：

```json
{
  "industry_type": "experiential",
  "digital_base": 12,
  "resource_readiness": 14,
  "pain_urgency": 22,
  "team_acceptance": 8,
  "compliance_risk": 10
}
```

**执行命令**：
```bash
python3 scripts/calculate_score.py -i score_input.json
```

**评分逻辑**：
1. 按行业类型加载动态权重（会销行业团队接受度权重更高=20%）
2. 每个维度归一化到百分制
3. 应用权重计算加权总分
4. 对照结论映射表输出等级和建议

### 2.2 输出结果

#### 五维度评分详情

| 维度 | 原始分 | 满分 | 权重（会销行业） | 加权得分 |
|------|--------|------|-----------------|---------|
| 数字化基础 | 12/25 | 25 | 25% | 12.0 |
| 资源准备度 | 14/20 | 20 | 15% | 10.5 |
| 痛点紧迫度 | 22/25 | 25 | 25% | 22.0 |
| 团队接受度 | 8/15 | 15 | 20% | 10.7 |
| 合规风险 | 10/15 | 15 | 15% | 10.0 |
| **总分** | - | 100 | - | **65.2** |

#### 诊断结论

| 项目 | 内容 |
|------|------|
| **总分** | **65.2 / 100** |
| **等级** | **B级** |
| **结论** | 有一定基础，建议先补短板 |
| **行动建议** | 补齐数据/团队短板后，再启动AI项目 |

#### 各维度简评

- **数字化基础（12/25）**：有CRM和企微等基础系统，但数据质量差、无数据中台，处于"有系统但用不好"的阶段
- **资源准备度（14/20）**：预算充足（80-120万），但IT人力薄弱（仅1人），管理层支持度高
- **痛点紧迫度（22/25）**：人力成本高、复购率下降、服务质量差等痛点突出，AI解决的匹配度高
- **团队接受度（8/15）**：员工平均年龄偏大、数字化素养偏低，是主要短板
- **合规风险（10/15）**：保健品行业合规要求高，营销话术、客户数据等都有合规风险

### 2.3 改进点说明

| 对比维度 | 纯Prompt版本 | v1.8.0 脚本化版本 |
|---------|-------------|------------------|
| **评分逻辑** | LLM凭感觉打分，每次结果不同 | **脚本化计算**，公式透明，结果可复现 |
| **行业适配** | 通用打分，不区分行业 | **5种行业动态权重**，会销行业团队接受度权重20%（默认15%） |
| **评分体系** | 简单的1-5星或百分比 | **5维度×满分不等×动态权重**，精细化建模 |
| **结论映射** | LLM临场组织语言 | **5级结论映射表**（S/A/B/C/D），口径统一 |
| **可维护性** | 改prompt容易出bug | **数据与逻辑分离**，修改权重改配置即可 |
| **可测试性** | 无法自动化测试 | **支持 --self-test 自测**，3个测试用例自动验证 |

> **【改进点】** 从"LLM凭经验打分"升级为"脚本化动态权重评分"。核心优势：
> 1. **结果可复现**：同样的输入永远得到同样的输出
> 2. **行业差异化**：5种行业类型权重不同，评分更精准
> 3. **逻辑透明**：每个分数怎么来的清清楚楚，方便向客户解释
> 4. **可测试**：内置自测功能，修改后一键验证

{

3. 百分制加权汇总，排序输出

- 提升复购直接带来营收增长 → ROI明确

### 3.3 改进点说明

| 对比维度 | 纯Prompt版本 | v1.8.0 脚本化版本 |
|---------|-------------|------------------|
| **场景数量** | LLM想到几个说几个，不固定 | **固定10大场景**，全覆盖评估 |
| **排序依据** | LLM主观推荐，说不清为什么 | **4维度加权打分**（痛点35%+ROI30%+技术15%+难度20%） |
| **动态调整** | 不具备，全靠LLM临场发挥 | **12项动态调整因子**，根据企业特征精准调整 |
| **可解释性** | "我觉得这个适合你"，无依据 | 每个场景每项分数透明，**可追溯为什么排第一** |
| **一致性** | 同一个企业问两次，推荐可能不同 | **结果确定性**，同样输入永远同样输出 |
| **基准数据** | 无 | **场景基准表**（基准分+适配行业+替代比例+成本区间） |

> **【改进点】** 从"LLM主观推荐"升级为"多维度动态评分排序"。核心优势：
> 1. **可解释性强**：每个推荐都能说出"为什么是这个"，客户更容易信服
> 2. **动态适配**：12项调整因子根据企业实际情况调整，比固定推荐精准
> 3. **全量评估**：10个场景全部打分排序，不会遗漏好的机会
> 4. **持续优化**：新增调整因子只需改脚本，不用改prompt

## Step 4：ROI三档测算

### 4.1 执行过程

场景推荐完成后，调用 `scripts/calculate_roi.py` 进行ROI三档测算。

**输入数据**：

```json
{
  "team_structure": {
    "customer_service": {"count": 25, "avg_salary": 5500},
    "content": {"count": 6, "avg_salary": 7000},
    "operations": {"count": 20, "avg_salary": 6500},
    "sales": {"count": 80, "avg_salary": 6000},
    "admin_hr": {"count": 8, "avg_salary": 5500}
  },
  "budget": 1000000,
  "team_acceptance": "low",
  "digital_base": "medium"
}
```

**执行命令**：
```bash
python3 scripts/calculate_roi.py -i roi_input.json
```

**计算逻辑**：
1. 计算各团队年人力成本
2. 按岗位AI替代比例（6个月/2年两档）计算年节省
3. 三档测算（保守/中性/乐观），考虑隐性成本和效果系数
4. 团队接受度×数字化基础整体调整系数
5. 两种策略对比（消灭岗位 vs 优化流程）

### 4.2 输出结果

#### 人力成本基线

| 岗位 | 人数 | 平均薪资 | 年人力成本 |
|------|------|---------|-----------|
| 客服售后 | 25人 | 5,500元 | 165万 |
| 内容创作 | 6人 | 7,000元 | 50.4万 |
| 运营/数据 | 20人 | 6,500元 | 156万 |
| 销售支持 | 80人 | 6,000元 | 576万 |
| 行政人力 | 8人 | 5,500元 | 52.8万 |
| **合计** | **139人** | - | **1,000.2万** |

> 注：一线门店员工181人未计入AI替代计算（主要是面对面服务，AI替代空间有限）

#### AI节省预估

| 时间节点 | 年节省金额 | 占人力成本比例 | 说明 |
|---------|-----------|--------------|------|
| 6个月时 | 277.3万/年 | 27.7% | 初步见效，替代简单重复性工作 |
| 2年成熟时 | 509.4万/年 | 50.9% | 深度应用，替代复杂决策辅助工作 |

#### 三档回本周期

| 档位 | 回本周期 | 总投入 | 年节省金额 | 效果系数 | 隐性成本 |
|------|---------|--------|-----------|---------|---------|
| 🔴 保守档 | **9.4个月** | 140万 | 178.3万 | 0.5x | 40% |
| 🟡 中性档 | **4.2个月** | 125万 | 356.6万 | 1.0x | 25% |

- 避免了"LLM做数学题"的不靠谱问题

**3. 领域知识结构化（本体知识库）**
- 8大知识模块统一管理在 ontology.py
- 行业配置、维度定义、场景基准、ROI参数、合规红线、坑点清单、校验规则、路线图模板

**4. 状态持久化**
- SessionState 支持保存为JSON，中断续答
- 企业咨询可能分多次完成，不用每次从头来
